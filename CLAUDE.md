# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

This started as "a scratch repo for trying out Claude Code workflows" (see README.md) but now contains a real two-part app: a personal budgeting/expense tracker.

- `server/` ‚Äî Express + better-sqlite3 backend (single-file SQLite DB)
- `client/` ‚Äî React + Vite frontend

The two are independent npm projects; there is no root `package.json` or monorepo tooling. Run/build each from its own directory.

## Commands

### Server (`server/`)
```
npm install
npm run dev          # node --watch src/index.js, listens on http://localhost:4000
npm start            # node src/index.js (no watch)
npm run smoke-test    # node src/scripts/smokeTest.js ‚Äî fetch-based end-to-end check against a running server
```
There is no automated test framework wired up; `smokeTest.js` is the only verification script. It expects the server to already be running on port 4000 and exercises the full transaction/transfer/summary flow via real HTTP calls, asserting on response values. Run it manually after starting `npm run dev`.

The SQLite DB file (`server/budget.db`, plus `-wal`/`-shm`) is gitignored and created automatically on first run via the migration in `server/src/migrations/001_init.sql`. Delete it to reset to a clean state (server re-applies the migration on next boot, including seeding the two accounts).

`server/.env.example` documents every server env var with a placeholder/comment and is safe to commit (root `.gitignore` ignores `server/.env`/`server/.env.*` but explicitly re-includes `server/.env.example`). Required vars: `TURSO_DATABASE_URL`/`TURSO_AUTH_TOKEN` (DB connection), `API_TOKEN` (static gate, sent as `X-API-Token`), `CORS_ORIGIN`, `OLLAMA_CLOUD_*` (LLM import-suggest, optional ‚Äî feature is hard-off if unset), and **`JWT_SECRET` (required, no default)** ‚Äî user-auth's signing secret; `server/src/services/authService.js` throws at module load if it's unset, which crashes the process on boot (every route 500s, not just auth ones), so this must be set before starting the server at all once the auth feature is present. Generate a strong value (e.g. `openssl rand -base64 32`); rotating it later invalidates every existing user's session at once (no per-session revocation exists).

### Client (`client/`)
```
npm install
npm run dev           # Vite dev server on http://localhost:5173, proxies /api/* to http://localhost:4000 (see vite.config.js)
npm run build          # production build to client/dist
npm run lint            # eslint .
npm run preview         # serve the production build locally
```
No test suite is configured for the client either ‚Äî `npm run build` is the main compile-correctness check.

To run the full app locally: start the server (`server/`, port 4000) and the client dev server (`client/`, port 5173) in separate terminals; the Vite proxy forwards API calls.

## Architecture

### Data model (single source of truth: `server/src/migrations/001_init.sql`, `002_budgets.sql`, `003_categories.sql`, `004_category_accounts.sql`, `006_users.sql`‚Äì`009_budgets_userid.sql`)
Two fixed accounts (`Spending` id=1, `Savings` id=2, seeded by the migration ‚Äî see `server/src/constants/categories.js` `ACCOUNTS` and the client's `client/src/constants/categories.js` `ACCOUNTS`/`ACCOUNT_NAMES`, which must stay in sync with the DB row ids). All money movement is one `transactions` table:
- `amount` is always stored positive; `direction` (`'in'` | `'out'`) carries the sign.
- Internal transfers (Spending ‚Üî Savings) are represented as **two linked rows**, not one: an `out` leg on the source account and an `in` leg on the destination account, both `is_transfer=1`, each pointing at the other via `linked_transaction_id`. This pairing is created/edited/deleted atomically in `server/src/services/transactionService.js` using a `better-sqlite3` transaction. When deleting a transfer pair, `linked_transaction_id` must be nulled on both rows *before* either row is deleted, or the self-referencing foreign key constraint fails.
- Categories live in a `categories` table (`server/src/migrations/003_categories.sql`, extended by `004_category_accounts.sql`) split into `outgoing`/`incoming` lists, and are **independent per account**: each account has its own editable set of categories and its own color palette, scoped by an `account_id` column (`004_category_accounts.sql` adds it, backfills existing rows to Spending, then clones them into an independent Savings set, and replaces the old global unique index with one scoped per account ‚Äî `idx_categories_account_name_list ON categories(account_id, lower(name), list)`). `transfer-in`/`transfer-out` are the only categories that stay system-managed and account-independent (`account_id IS NULL`, `is_system=1`); the plain `POST /transactions` endpoint rejects them (only `POST /transactions/transfer` may set them). `services/categoryService.js` is the only module that reads/writes the table; `routes/categories.js` exposes it (`GET /api/categories?account_id=`, `POST /api/categories`, `DELETE /api/categories/:id`).
- Running balance per account and daily/monthly aggregates are **computed on read via SQL**, never stored redundantly ‚Äî see the window function in `server/src/services/balanceService.js` (`SUM(...) OVER (PARTITION BY account_id ORDER BY date, id)`) and the `GROUP BY` queries in `server/src/services/summaryService.js`. Monthly category breakdowns exclude transfers (`is_transfer = 0`) since they're internal movement, not real income/spend.
- A second table, `budgets` (one row per user+month+category, `UNIQUE(user_id, month, category)`), holds per-category monthly budget amounts: `month` is a `'YYYY-MM'` string, `category` is one of Spending's outgoing categories (budgeting is **Spending-only** ‚Äî `budgetService.js` validates via `getBudgetableNames(ACCOUNTS.SPENDING, userId)`, a Savings category of the same name is never a valid budget target), `amount REAL NOT NULL CHECK (amount >= 0)`. Every Spending outgoing category always has a budget value as far as any caller can observe ‚Äî **the unset-vs-zero distinction was removed from the product/API surface**: a category nobody has ever set behaves identically to one explicitly set to `amount = 0`. Internally the table stays **sparse** (a row exists only once a `PUT` happens ‚Äî there was no migration and no pre-seeding of zero rows); `getBudgetsForMonth` synthesizes the `0` default on read for any category with no row. So a row vs. no-row still physically differ at the DB level, but that difference is now purely an implementation detail ‚Äî no caller should query the table to infer "did the user set this?", since that question is no longer product-meaningful. `db.js` applies migrations via independent existence guards in sequence ‚Äî a table-existence check for `transactions` (`001_init.sql`), then `budgets` (`002_budgets.sql`), then `categories` (`003_categories.sql`), then a column-existence check (`pragma_table_info('categories')` for `account_id`) for `004_category_accounts.sql`, then four more guards for `006_users.sql`‚Äì`009_budgets_userid.sql` (below) ‚Äî rather than a single versioned migration runner; there is no migration-version-tracking table yet, so adding a further migration means adding a further guard in the same pattern.
- **Users and per-user data isolation** (`006_users.sql`‚Äì`009_budgets_userid.sql`, added for the user-auth feature): a `users` table (`id INTEGER PRIMARY KEY AUTOINCREMENT`, `username TEXT NOT NULL UNIQUE`, `password_hash TEXT` nullable ‚Äî NULL for guest accounts, `is_guest INTEGER NOT NULL DEFAULT 0`, `created_at TEXT NOT NULL DEFAULT (datetime('now'))`). `007_transactions_userid.sql` adds a plain `user_id INTEGER` column to `transactions` (plus `idx_transactions_user_date ON transactions(user_id, date, id)`); `008_categories_userid.sql` adds the same plain `user_id INTEGER` to `categories`, drops the old `idx_categories_account_name_list` and replaces it with `idx_categories_user_account_name_list ON categories(user_id, account_id, lower(name), list)` (system rows ‚Äî `is_system=1`, `transfer-in`/`transfer-out` ‚Äî stay `user_id IS NULL` permanently and globally, never claimed or per-user-seeded); `009_budgets_userid.sql` **rebuilds** `budgets` (SQLite/libSQL can't `ALTER` a table-level `UNIQUE` constraint) into a `budgets_new` with `user_id INTEGER` inserted after `id` and `UNIQUE(user_id, month, category)` replacing the old `UNIQUE(month, category)`, then renames it back ‚Äî every existing row's `id`/data is preserved, only `user_id` is backfilled as `NULL`. **Every `user_id` column is a plain `INTEGER` with no `REFERENCES`/FK** ‚Äî SQLite's `ALTER TABLE ADD COLUMN` cannot add a foreign key to an existing table ‚Äî so per-user data isolation is enforced entirely by a service-layer `WHERE user_id = ?` (or `AND user_id = ?`) discipline in every query that touches `transactions`, `budgets`, or non-system `categories`; there is no DB-level constraint or row-level security backstop. Any new query against those three tables that omits a `user_id` predicate is a bug, not a stylistic choice.
- **First-signup legacy claim + per-user category seeding** (`server/src/services/authService.js`'s `signup()`): `user_id IS NULL` on `transactions`/`budgets`/non-system `categories` means "pre-auth legacy data, unclaimed" (as opposed to a system category's permanent `user_id IS NULL`, which is distinguished by `is_system=1`). The **first non-guest signup** (checked via `SELECT COUNT(*) FROM users WHERE is_guest = 0`, evaluated *before* the new user is inserted, inside the same write transaction) claims all of it: `UPDATE transactions/budgets/categories SET user_id = :newId WHERE user_id IS NULL` (the `categories` update additionally requires `AND is_system = 0`). Guest signups never compute or consult this check, so a guest session can never consume the one-time claim. Every signup *after* the first (and every guest signup) instead calls `seedCategoriesForUser(userId, exec)` (`categoryService.js`), which inserts a fresh, independent 22-row category set (the same 11 outgoing/incoming categories from `003_categories.sql`'s seed, duplicated across both `ACCOUNTS.SPENDING`/`ACCOUNTS.SAVINGS`) driven by an in-code `SEED_CATEGORIES` constant ‚Äî not by cloning the (already-claimed) legacy NULL rows, since those no longer exist after the first claim.

### Backend structure (`server/src`)
`index.js` wires route groups under `/api`: `routes/accounts.js`, `routes/transactions.js` (CRUD + the special `/transactions/transfer` endpoint), `routes/summary.js` (`/summary/daily`, `/summary/monthly`), `routes/budgets.js`, `routes/categories.js`, `routes/imports.js`, and `routes/auth.js` (see below, not gated by `requireUser`). Routes delegate to `services/` (`transactionService`, `balanceService`, `summaryService`, `budgetService`, `categoryService`, `authService`), which are the only modules that touch `db.js` directly. `transactionService.js`, `budgetService.js`, and `authService.js` each export their own `ValidationError` class that routes map to HTTP 400 (`authService.js` also exports an `AuthenticationError` mapped to 401). Every service call from every route now takes `req.userId` as an explicit first-class argument (see the auth section below) ‚Äî this threading is additive to the existing explicit-parameter style already used for the async `@libsql/client` `.execute({sql, args})` calls.

- **Authentication** (`routes/auth.js` ‚Üí `services/authService.js`, `middleware/requireUser.js`): two orthogonal gates in `index.js`, checked in order ‚Äî (1) a **static API-token gate**, `X-API-Token: <API_TOKEN>` header (moved off `Authorization` to free that header for the user JWT; the `?token=<API_TOKEN>` query fallback on `GET /transactions/export` is unchanged) ‚Äî infra-level "can you talk to this API at all," skipped entirely if `API_TOKEN` is unset; then (2) `requireUser`, mounted as router-level middleware on every router *except* `/api/auth` (`accounts`, `transactions`, `summary`, `budgets`, `categories`, `imports` ‚Äî `imports` gets `requireUser` before its own `express.json({limit:'25mb'})`), which is identity ‚Äî "who are you." Both gates pass through `OPTIONS` preflight requests untouched.
  - `requireUser` reads the user JWT from `Authorization: Bearer <jwt>`, or ‚Äî for `GET /api/transactions/export` *only* (`isExportRoute()`: `req.method==='GET' && req.baseUrl==='/api/transactions' && req.path==='/export'`; anchor-nav downloads can't set headers) ‚Äî from a `?authToken=<jwt>` query param. This query-param path is deliberately scoped to that one route: it was originally accepted on every route, which a security review flagged (a non-expiring JWT in a URL query string is a durable-credential-in-logs exposure on any GET), and was narrowed to the export route only as a follow-up fix. On success, `req.userId = payload.sub` and `req.user = payload` are set and every downstream service call is scoped by `req.userId`; on missing/invalid/expired token, `401 { error: 'Authentication required' }`, no fallthrough.
  - JWT scheme: `jsonwebtoken`, algorithm **HS256** (pinned on verify ‚Äî `{ algorithms: ['HS256'] }` ‚Äî blocking `alg:none`/algorithm-confusion attacks), claims `{ sub: <userId>, username, isGuest }`, **no `exp` claim** ‚Äî tokens do not expire; the only revocation lever is rotating `JWT_SECRET`, which invalidates every issued token at once (logs everyone out ‚Äî treat as a "communicate before doing this in prod" operation, not a routine config tweak). Secret: `JWT_SECRET` env var, read once at `authService.js` module load; the module **throws immediately if it's unset**, and since both `routes/auth.js` and `middleware/requireUser.js` import it unconditionally at boot, an unset `JWT_SECRET` crashes the whole process on cold start (every route 500s, not just auth ones) rather than degrading gracefully. Passwords are hashed with `bcryptjs` (pure-JS, no native bindings ‚Äî deliberately chosen over native `bcrypt` for serverless-build safety) at cost factor 10; `signup()` additionally enforces an 8-char minimum and a 72-byte maximum (bcrypt's silent-truncation boundary, checked via `Buffer.byteLength`, not `.length`, so multi-byte passwords count correctly).
  - `POST /api/auth/signup` `{ username, password }` ‚Üí **201** `{ token, user: { id, username, isGuest: false } }` (400 on missing fields or a taken username); runs the first-signup-claim sequence from the data-model section above inside one write transaction. `POST /api/auth/login` `{ username, password }` ‚Üí **200** `{ token, user }`; **401** on any bad credentials *or* a guest username (guests have `password_hash NULL`, so a guest can never log back in) ‚Äî the not-found/guest branch runs a dummy `bcrypt.compare` against a fixed hash before rejecting, so it costs about the same as a real wrong-password comparison (closes a login-timing user-enumeration side channel; the rejection message is identical either way). `POST /api/auth/guest` `{}` ‚Üí **201** `{ token, user: { id, username: 'guest_<uuid>', isGuest: true } }`, always seeds a fresh isolated category set, never claims legacy data or consumes the first-signup claim. `GET /api/auth/me` (has `requireUser` applied *inside* the router, not router-level) ‚Üí **200** `{ user: { id, username, isGuest } }` looked up by `req.userId`; **401** if the token is missing/invalid ‚Äî used for session-check-on-load. `POST /api/auth/logout` ‚Üí **200** `{ ok: true }` always; stateless no-op (no server-side session to invalidate) ‚Äî logout is purely a client-side token discard.
  - **Every service is now userId-scoped**: `transactionService.js` stamps `user_id` on every INSERT (both legs of a transfer) and adds `AND user_id = :userId` to every existence-check SELECT and every UPDATE/DELETE (another user's transaction id reads as a 404, never mutates); `deleteAllTransactions` is `WHERE user_id = :userId`, never a global wipe. `balanceService.js` filters `user_id` *inside* the running-balance window subquery, not as an outer filter, so another user's rows never enter the `PARTITION BY account_id` computation; the `accounts` table itself stays global (Spending/Savings are shared row ids, not per-user). `summaryService.js` scopes every aggregate query by `user_id`, with the per-account daily `LEFT JOIN`'s predicate in the `ON` clause (not the `WHERE`) so a user's zero-activity account still produces a row. `budgetService.js` scopes `getBudgetsForMonth` by `user_id` and upserts on `ON CONFLICT(user_id, month, category)`. `categoryService.js` is the one exception with two branches: system rows (`is_system=1`, `transfer-in`/`transfer-out`) are global and readable by everyone (`user_id IS NULL` forever), so most reads scope `AND user_id = :userId` (equivalent to the fuller `(user_id = :userId OR is_system = 1)` once combined with an existing `is_system = 0` filter) ‚Äî except `deleteCategory`'s lookup SELECT, which does use the literal `(user_id = :userId OR is_system = 1)` form, so that a request to delete a system category still resolves to "found, and rejected as a system category" (400) rather than a false 404; the actual `DELETE` statement stays a strict `WHERE id = :id AND user_id = :userId`, so it can never touch another user's or a system row. Writes (`createCategory`) always stamp `user_id` and `is_system = 0`. `importService.commitImport` threads `userId` into every row's `createTransaction`/`createCategory`/`createTransfer` call, and the import LLM-suggest route builds its `knownCategories` vocabulary from the *caller's own* categories only (no cross-user leak through the AI-suggest path).

- **Budgets API** (`routes/budgets.js` ‚Üí `services/budgetService.js`): budgeting applies to the **Spending account's categories only** ‚Äî there is no per-account budgeting. `GET /api/budgets?month=YYYY-MM` returns `{ month, budgets: [{ category, amount }] }`, scoped to the caller's `req.userId`, and the array is now **exhaustive**: every one of `getBudgetableNames(ACCOUNTS.SPENDING, userId)`'s categories is always present, in that function's order (not alphabetical), with `amount` defaulting to `0` for any category that has no row in the `budgets` table for this user ‚Äî there's no "unset" entry to omit anymore. `PUT /api/budgets` with body `{ month, category, amount }` upserts a single category's budget for the caller (`INSERT ... ON CONFLICT(user_id, month, category) DO UPDATE`) and returns the saved `{ month, category, amount }`; `amount = 0` is and always was a valid, explicit value. There is no `DELETE /api/budgets` endpoint ‚Äî it was removed along with the service's `clearBudget` and the client's "Clear" button, since "revert to unset" is no longer a concept consumers can act on; to zero out a category the user sets its budget to `0` instead. Both endpoints validate `month` against `/^\d{4}-(0[1-9]|1[0-2])$/`, `category` against `getBudgetableNames(ACCOUNTS.SPENDING)` (Spending's live outgoing category names), and (for `PUT`) `amount` as a finite number `>= 0`; failures throw `ValidationError` ‚Üí HTTP 400. There is no dedicated "budget vs. actual" endpoint ‚Äî the comparison against real spend is done **client-side**, and lives in exactly one place: the Overview dashboard's Budget card (`pages/DashboardPage.jsx`), which derives actuals from the already-fetched, Spending-filtered transactions array rather than calling `/api/summary/monthly` ‚Äî see the Overview budget card note below. `pages/BudgetPage.jsx` is edit-only now (no comparison chart) but still calls both `GET /api/budgets` and `GET /api/summary/monthly` to show each row's current-month actual spend next to its editable budget input.

- **Import LLM-suggest endpoint** (`routes/imports.js` ‚Üí `services/importLlmService.js`): `POST /api/imports/suggest` is ASSIST-only ‚Äî it never replaces or shortcuts the deterministic import pipeline (`parseFile` ‚Üí Step 2-3 mapping ‚Üí Step 4 review ‚Üí `commitImport`'s server-side re-validation), it only produces better defaults for the same `columnMapping`/`categoryMapping`/`accountMapping` prefill seams `guessColumnMapping.js` and the fuzzy matchers already fill. Request body: `{ headers, sampleRows, accountLabels, commentCol, fileHash }` (JSON, not multipart ‚Äî the file was already parsed by `/parse`); the server builds `knownCategories` itself live from `categoryService.getOutgoingNames/getIncomingNames` across both accounts (not caller-supplied, so a client can't inject a fake vocabulary the LLM could "satisfy"). Response is always HTTP 200 with `{ suggestion: null }` for every expected failure (key unset, network/timeout, non-2xx upstream, malformed/oversized body, schema-invalid) ‚Äî a real 500 only happens for a genuinely unexpected server error. `importLlmService.js` is the only module that talks to Ollama Cloud; its `buildRequest`/`parseResponse` are deliberately isolated from validation/caching so the wire shape can be corrected in one place. **The wire shape is an ASSUMPTION, not a confirmed contract**: `buildRequest` posts an OpenAI chat-completions-compatible body (`POST {OLLAMA_CLOUD_BASE_URL}/v1/chat/completions`, `Authorization: Bearer {OLLAMA_CLOUD_API_KEY}`, `{ model, messages, response_format: { type: 'json_object' } }`) and `parseResponse` unwraps `choices[0].message.content` as a JSON string ‚Äî this has not been verified against a real Ollama Cloud call (only mocked transports per the build dispatch); expect to revisit these two functions after the first real-API test. Config: `OLLAMA_CLOUD_API_KEY` (bearer token; feature is hard-off if unset/blank), `OLLAMA_CLOUD_MODEL` (model name string), `OLLAMA_CLOUD_BASE_URL` (must parse as `https:` or the feature is treated as not-configured ‚Äî no unauthenticated or default-endpoint fallback). `readConfig()` reads `process.env` fresh on every call (not cached at module load). Server enforces, regardless of what the caller sends: a hard 5-row sample cap (`MAX_SAMPLE_ROWS`, not caller-raisable), comment-column redaction keyed off the caller-supplied `commentCol` index (cells replaced with the literal `<redacted>` before the prompt is built), a request timeout (`REQUEST_TIMEOUT_MS`, `AbortController`-based), and a response-size cap (`MAX_RESPONSE_BYTES`) with clean rejection of non-JSON/oversized bodies. `validateSuggestion` is strict allow-list validation (column indices in range, enums checked, category names checked against `knownCategories` unless `isNew`, account ids in `{1,2}`) with **drop-whole-on-any-deviation** semantics ‚Äî there's no partial-trust/field-repair path. Results are cached in-memory by `fileHash` (caller-supplied, preferably a hash of the full uploaded file's bytes; the route derives a sha256 fallback from `headers`+`sampleRows` if omitted) in a process-lifetime `Map` with no eviction ‚Äî re-clicking "Suggest" on an unchanged file is free and skips the network entirely. `dotenv/config` is now loaded as the first import in `server/src/index.js` (the server had no `.env` support before this feature); `server/.env` and `server/.env.*` are gitignored (root `.gitignore`) ‚Äî there is no `.env.example` checked in yet, so a fresh clone needs the three `OLLAMA_CLOUD_*` vars documented/communicated out-of-band until one is added.

### Frontend structure (`client/src`)
Three routed pages (`react-router-dom`) under `App.jsx`: `pages/DashboardPage.jsx`, `pages/TransactionsPage.jsx`, `pages/BudgetPage.jsx` (nav order in `components/layout/Header.jsx`: Overview, Transactions, Budget ‚Äî there is no standalone Breakdown page/route anymore; its content now lives at the bottom of Overview, see below). All server I/O goes through `api/client.js` (a thin fetch wrapper hitting `/api/...`, proxied by Vite in dev), including `getBudgets`/`setBudget` (there is no `clearBudget` ‚Äî that call and its "Clear" button were removed along with `DELETE /api/budgets`) and `api.signup/login/guest/me/logout` (see auth below).

- **Auth gating and session management** (`contexts/auth.js` + `contexts/AuthContext.jsx`, `components/auth/`): `AuthProvider` wraps the whole tree (outside `ThemeProvider`'s children but the auth screen itself still renders inside `ThemeProvider` so it's themed). On mount it reads `localStorage['ledger.authToken']`; if present, calls `GET /api/auth/me` to verify it's still valid (200 ‚Üí `setUser`; failure ‚Üí clears the token) ‚Äî this is the "session-check-on-load." `App.jsx`'s internal `AppShell` reads `useAuth()`: `loading` ‚Üí a minimal splash; no `user` ‚Üí the full-page `AuthScreen` (login/signup tabs + "Use as guest," no header/nav ‚Äî replaces the app rather than overlaying it); `user` present ‚Üí the normal routed app. Because categories and transaction-activity data are now per-user, `<CategoriesProvider key={user.id}>` and `<TransactionActivityProvider key={user.id}>` are keyed by the current user's id so switching accounts **remounts** both providers and refetches user-scoped data instead of leaking the previous user's state across the switch.
  - **Storage**: the active session's JWT lives in `localStorage['ledger.authToken']` (read fresh on every `api/client.js` call, not cached at module load, so an account switch takes effect on the very next request with no reload); `localStorage['ledger.authSessions']` is a `{ [username]: token }` JSON map of every session this browser has ever logged into, used to populate the account-switcher's "Switch account" list without re-authenticating.
  - **`AuthContext`** exposes `{ user, token, loading, sessions, login, signup, guest, logout, switchAccount, addAccount }`. `login`/`signup`/`guest` all funnel through one `applySession(token, user)` helper that sets the active token, records it into `ledger.authSessions`, and sets `user`. `switchAccount(username)` promotes a stored session's token to active and **re-verifies it** via `GET /api/auth/me` (a stored token can go stale, e.g. after a `JWT_SECRET` rotation) ‚Äî on failure it drops that session and clears the token rather than silently appearing "logged in." `logout` calls `POST /api/auth/logout` best-effort (the server call is a stateless no-op; a network failure there is swallowed) and then does the client-side token discard that actually ends the session. Any 401 response from **anywhere** in the app (not just calls `AuthContext` made directly) triggers the same drop-to-auth-screen behavior via a `window` `CustomEvent('ledger:unauthorized')` dispatched by `api/client.js`'s `request()`/`requestFormData()` (excluding 401s from `/auth/*` itself, which are ordinary inline login/signup form errors, not a dead-session signal) ‚Äî `AuthContext` listens for this event globally.
  - **UI**: `components/auth/AuthScreen.jsx` (full-page, tab-switched login/signup + guest button with an isolation warning that a guest session can't be logged back into once its browser storage is cleared), `components/auth/AuthForms.jsx` (shared `LoginForm`/`SignupForm`, reused by both `AuthScreen` and `AddAccountModal` so there's exactly one place owning field state/validation), `components/auth/AddAccountModal.jsx` (same modal/portal-to-`#modal-root` convention as `TransactionModal.jsx`; submitting either tab promotes that account to active ‚Äî the `key={user.id}` remount in `App.jsx` is what actually performs the "switch"), and `components/layout/AccountSwitcher.jsx` (a Header dropdown showing the current username, a "Switch account" list built from `sessions`, "Add account," and "Log out").
  - **`api/client.js`'s dual-header scheme**: `authHeader()` attaches **both** `X-API-Token: <API_TOKEN>` (the static token, if `VITE_API_TOKEN` is set ‚Äî moved off `Authorization` to free that header) and `Authorization: Bearer <ledger.authToken>` (if a session is active) to every request; neither implies the other. The export download (`components/transactions/ExportModal.jsx`, anchor-nav, can't set headers) appends both as query params instead: `?token=<API_TOKEN>&authToken=<jwt>`, matching `requireUser`'s export-only query fallback on the server.

Years/months/days display most-recent-first; transactions *within* a day keep the array's original (chronological) order. `DayGroup` computes per-day combined and per-account in/out subtotals plus end-of-day balances (`dayTotals.js`) and only renders the per-account breakdown when a day actually spans more than one account (avoids redundant rows on a single-account day or filtered view).

For non-transfer `out` transactions it returns `'highlight-orange'`/`'highlight-red'` (`>$20`/`>$40`), rendered in `TransactionRow.jsx` as a small colored dot next to the amount, not an amount-cell background fill. `transferBadgeFor` provides the direction cue instead: an IN/OUT text badge (green/red) rendered in the comment cell, so the in/out distinction isn't color-only (WCAG 1.4.1).

Invalid input (negative, non-numeric, empty) on commit reverts to the last valid value rather than crashing or saving garbage. ## Known security risks

- **`xlsx` (SheetJS) `@0.18.5`, npm registry ‚Äî accepted, unpatched advisories.** Used by `server/src/services/importService.js`'s `parseFile` to parse uploaded CSV/.xlsx bytes for the transaction import feature. `npm audit` reports two unpatched high-severity advisories with "no fix available" on npm: Prototype Pollution (GHSA-4r6h-8v6p-xvw6) and ReDoS (GHSA-5pgg-2g8v-p4x9) ‚Äî SheetJS only publishes patched builds via their own CDN (cdn.sheetjs.com), not the npm registry. **Accepted as-is** because this is a single-user local app: the only untrusted input reaching `xlsx` is the user's own uploaded file, processed against their own local DB ‚Äî there is no multi-tenant or network-exposed attack surface, and the advisories grant no privilege beyond what the local user already has. **Reopens as a blocker** if any of: (1) the app becomes multi-user or network-exposed, (2) the import endpoint ever ingests files from anyone but the local user, or (3) a registry fix becomes available (re-run `npm audit` periodically to check). Exit path at that point: pin SheetJS's CDN-hosted patched build, or move to a CSV-only import path via a maintained CSV parser.

- **LLM-assisted import ("Suggest with AI") ‚Äî first and only path by which app data leaves the user's machine to an external third party (Ollama Cloud).** Every other feature in this app (parse/map/review/commit, the rest of the import wizard included) runs 100% locally against the user's own SQLite file; this is the one deliberate exception, and it changes the threat model for the data it touches, so the conditions below are what keep it acceptable:
  - **Opt-in, per-file, with a mandatory consent/preview step ‚Äî not on by default, not a one-time global toggle.** Nothing is sent until the user clicks "Suggest with AI" (`client/src/components/imports/ImportSuggestAI.jsx`, Step 2 of the import wizard) *and* then explicitly clicks "Confirm and send" on a panel that names Ollama Cloud as an external third party and renders the literal payload about to be sent (not an approximation ‚Äî the preview is built by the same `buildPayload()` function that constructs the real request body).
  - **Minimal payload, server-enforced regardless of what the client sends:** headers, up to 5 sample rows (`MAX_SAMPLE_ROWS` in `server/src/services/importLlmService.js`, hard-capped server-side and not caller-raisable even if the client sent more), the comment/memo column redacted to the literal string `<redacted>` before the prompt is built (keyed off the client-supplied `commentCol` index ‚Äî both client preview and server payload redact the same way, from the same source value, so they can't drift), and the app's own category vocabulary (`knownCategories`, built server-side from `categoryService`, not client-supplied). No full file, no full transaction history, no API key, ever leaves in the request.
  - **The API key (`OLLAMA_CLOUD_API_KEY`) is strictly server-side.** Read only in `server/src/services/importLlmService.js` via `process.env`, never returned in any response, never logged (`console.error` paths in `routes/imports.js`'s `handleError` stay generic), and never shipped to the browser (not a `VITE_`-prefixed var, not in any client bundle). The feature is hard-disabled ‚Äî `isConfigured()`/`readConfig()` return falsy and `POST /api/imports/suggest` resolves `{ suggestion: null }` ‚Äî whenever `OLLAMA_CLOUD_API_KEY`, `OLLAMA_CLOUD_MODEL`, or `OLLAMA_CLOUD_BASE_URL` is unset/blank, or the base URL isn't `https:`; there is no unauthenticated or default-endpoint fallback.
  - **The LLM's response is strictly schema-validated, drop-whole on any deviation.** `validateSuggestion` in `importLlmService.js` allow-lists every field (column indices in range, fixed enums, category names checked against the server-built vocabulary unless explicitly flagged `isNew`, account ids in `{1,2}`); any single invalid field discards the entire suggestion rather than partially trusting it. The response only ever **prefills** the same `columnMapping`/`categoryMapping`/`accountMapping` state the deterministic guessers (`guessColumnMapping.js`, fuzzy matchers) already populate ‚Äî it never reaches the commit path directly. The existing Step 4 review gate (every flagged row surfaced, `commitImport`'s server-side re-validation of every row inside one atomic transaction) is unchanged and remains the actual integrity boundary; the LLM suggestion is exactly as overridable, and exactly as easy for the user to ignore or correct, as a wrong fuzzy-match guess is today.
  - **Any failure mode degrades silently to the pre-existing deterministic prefill.** Timeout (server-enforced via `AbortController`, `REQUEST_TIMEOUT_MS`), network error, non-2xx upstream response, oversized/non-JSON body, or schema-invalid content all collapse to the same `{ suggestion: null }` signal from the server; the client (`ImportSuggestAI.jsx`) treats that identically to a thrown fetch error ‚Äî `onResult(null)` is a no-op, the wizard's existing `guessColumnMapping` + fuzzy category/account prefill is left completely untouched, and a non-alarming `role="status"` note is shown instead of an error. The wizard is never blocked on this call.
  - **Accepted as-is** under the current threat model: single-user, local app, opt-in per file, minimal/redacted payload, no auto-commit. **Reopens for re-review** if any of: (1) the app ever becomes multi-user or network-exposed (egressing one user's data on another's action is a materially different risk), (2) auto-commit or any "auto-apply AI suggestion without review" path is ever added (this would remove the only backstop that currently contains a maximally-wrong LLM response to "a bad prefill the user must still catch," not a bad write), (3) the per-file opt-in/preview gate is ever weakened to an always-on default or a buried settings toggle, or (4) the sample-row cap, redaction, or schema-validation drop-whole behavior is loosened.

- **User-auth model (JWT + service-layer isolation) ‚Äî reviewed, no Critical/High findings; a few accepted tradeoffs and two fixed Mediums.**
  - **JWT stored in `localStorage`, readable by any injected script (XSS exposure).** Accepted per the auth contract: this is a client-rendered SPA hitting a separate API origin with no cookie-based session to protect instead, so there's no lower-risk storage option available without adding cookie/CSRF machinery this app doesn't otherwise need. **Reopens** if the app ever gains a plausible XSS vector (e.g. rendering unsanitized third-party/imported HTML) that this token theft would meaningfully compound.
  - **Tokens do not expire (no `exp` claim).** Deliberate, per the contract ("stays signed in until explicit logout"); the only revocation lever is rotating `JWT_SECRET`, which invalidates every issued token for every user simultaneously ‚Äî an intentional, coarse, all-or-nothing control, not a bug. Combined with the localStorage-XSS point above, a stolen token is a permanent credential until that rotation happens. **Reopens as worth revisiting** if the app is deployed somewhere a leaked token has higher consequence than "one personal-budget dataset" ‚Äî the fix on the table (not yet built) is a bounded `exp` (e.g. 30‚Äì90 days) with silent client-side refresh.
  - **No rate limiting on `/api/auth/*` (login/signup/guest).** Accepted for now: the auth router still sits behind the static `X-API-Token` gate (so it's not reachable by the open internet without that token first), and bcrypt's cost-10 hashing already slows brute-force attempts considerably. **Reopens as a blocker before any deploy where the static token itself is no longer an effective secret** (e.g. it leaks, or the app goes multi-tenant with different trust levels per client) ‚Äî a serverless deployment also has no shared in-memory state for a naive rate limiter, so an effective fix would need an external store (Redis/Turso-backed counter), not just `express-rate-limit`'s default in-memory bucket.
  - **The `?authToken=<jwt>` query-param fallback was originally accepted on every route, then narrowed to `GET /api/transactions/export` only.** A security review found the original (contract-literal) implementation of `requireUser` honored `?authToken` on any route, which ‚Äî combined with the non-expiring JWT above and serverless platforms logging full request URLs including query strings ‚Äî meant a working, permanent credential could land in access logs, proxy logs, browser history, or a `Referer` header from an ordinary API call, not just the export download it was meant for. **Fixed**: `middleware/requireUser.js`'s `isExportRoute()` check now gates the query fallback to exactly `GET /api/transactions/export`; every other route only accepts `Authorization: Bearer <jwt>`. The equivalent static-token `?token=` fallback on `index.js`'s gate remains unscoped (accepted ‚Äî it's shared infra-level access, not a per-user identity credential, so the same log-exposure argument applies with lower severity).
  - **Login timing/enumeration.** Fixed: `login()` runs a dummy `bcrypt.compare` against a fixed hash on the "user not found or is a guest" branch before rejecting, so that branch costs about the same as a real wrong-password comparison ‚Äî closes a timing side channel that let an attacker infer whether a username exists. `signup`'s distinct "username is already taken" 400 remains a direct existence oracle, accepted as a standard signup-UX tradeoff (not fixed, not planned to be).
  - **Password floor.** Fixed: `signup` (not `login`, which must keep accepting pre-existing short passwords) now rejects passwords under 8 characters or over 72 bytes (bcrypt's silent-truncation boundary), via the existing `ValidationError` ‚Üí 400 path.
  - **No DB-level foreign key or row-level security for `user_id` ‚Äî isolation is a service-layer discipline, not a database guarantee.** SQLite's `ALTER TABLE ADD COLUMN` cannot add a `REFERENCES` constraint to an existing table, so every `user_id` column (`transactions`, `budgets`, `categories`) is a plain `INTEGER` with no FK. The **entire** integrity boundary is every query in `transactionService.js`/`balanceService.js`/`summaryService.js`/`budgetService.js`/`categoryService.js` correctly adding a `user_id` predicate or stamp ‚Äî this was audited query-by-query (no Critical/High findings; every read/write scoped, every IDOR path 404s rather than mutates cross-user). **Accepted as the permanent design** (not a temporary gap ‚Äî there is no migration path to a real FK without a full table rebuild, and the service-layer discipline has been reviewed and holds); any *new* query added later against these three tables that omits a `user_id` clause is a regression, not a stylistic choice, and should fail review on sight.
  - **Open CORS by default if `CORS_ORIGIN` is unset in production.** `index.js` falls back to a fully-open `cors()` when the env var isn't set. Not a CSRF risk (auth is header/localStorage-based, no cookies), but combined with the client-bundle-visible static token, an open CORS policy means any origin that obtains the token can call the API. **Not a code-level fix** ‚Äî set `CORS_ORIGIN` to the deployed frontend's origin in the server's deploy environment; this is an infra/deploy-checklist item, not something the code needs to change to support.
