# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

This started as "a scratch repo for trying out Claude Code workflows" (see README.md) but now contains a real two-part app: a personal budgeting/expense tracker.

- `server/` — Express + better-sqlite3 backend (single-file SQLite DB)
- `client/` — React + Vite frontend

The two are independent npm projects; there is no root `package.json` or monorepo tooling. Run/build each from its own directory.

## Commands

### Server (`server/`)
```
npm install
npm run dev          # node --watch src/index.js, listens on http://localhost:4000
npm start            # node src/index.js (no watch)
npm run smoke-test    # node src/scripts/smokeTest.js — fetch-based end-to-end check against a running server
```
There is no automated test framework wired up; `smokeTest.js` is the only verification script. It expects the server to already be running on port 4000 and exercises the full transaction/transfer/summary flow via real HTTP calls, asserting on response values. Run it manually after starting `npm run dev`.

The SQLite DB file (`server/budget.db`, plus `-wal`/`-shm`) is gitignored and created automatically on first run via the migration in `server/src/migrations/001_init.sql`. Delete it to reset to a clean state (server re-applies the migration on next boot, including seeding the two accounts).

### Client (`client/`)
```
npm install
npm run dev           # Vite dev server on http://localhost:5173, proxies /api/* to http://localhost:4000 (see vite.config.js)
npm run build          # production build to client/dist
npm run lint            # eslint .
npm run preview         # serve the production build locally
```
No test suite is configured for the client either — `npm run build` is the main compile-correctness check.

To run the full app locally: start the server (`server/`, port 4000) and the client dev server (`client/`, port 5173) in separate terminals; the Vite proxy forwards API calls.

## Architecture

### Data model (single source of truth: `server/src/migrations/001_init.sql`, `002_budgets.sql`, `003_categories.sql`, `004_category_accounts.sql`)
Two fixed accounts (`Spending` id=1, `Savings` id=2, seeded by the migration — see `server/src/constants/categories.js` `ACCOUNTS` and the client's `client/src/constants/categories.js` `ACCOUNTS`/`ACCOUNT_NAMES`, which must stay in sync with the DB row ids). All money movement is one `transactions` table:
- `amount` is always stored positive; `direction` (`'in'` | `'out'`) carries the sign.
- Internal transfers (Spending ↔ Savings) are represented as **two linked rows**, not one: an `out` leg on the source account and an `in` leg on the destination account, both `is_transfer=1`, each pointing at the other via `linked_transaction_id`. This pairing is created/edited/deleted atomically in `server/src/services/transactionService.js` using a `better-sqlite3` transaction. When deleting a transfer pair, `linked_transaction_id` must be nulled on both rows *before* either row is deleted, or the self-referencing foreign key constraint fails.
- Categories live in a `categories` table (`server/src/migrations/003_categories.sql`, extended by `004_category_accounts.sql`) split into `outgoing`/`incoming` lists, and are **independent per account**: each account has its own editable set of categories and its own color palette, scoped by an `account_id` column (`004_category_accounts.sql` adds it, backfills existing rows to Spending, then clones them into an independent Savings set, and replaces the old global unique index with one scoped per account — `idx_categories_account_name_list ON categories(account_id, lower(name), list)`). `transfer-in`/`transfer-out` are the only categories that stay system-managed and account-independent (`account_id IS NULL`, `is_system=1`); the plain `POST /transactions` endpoint rejects them (only `POST /transactions/transfer` may set them). `services/categoryService.js` is the only module that reads/writes the table; `routes/categories.js` exposes it (`GET /api/categories?account_id=`, `POST /api/categories`, `DELETE /api/categories/:id`).
- Running balance per account and daily/monthly aggregates are **computed on read via SQL**, never stored redundantly — see the window function in `server/src/services/balanceService.js` (`SUM(...) OVER (PARTITION BY account_id ORDER BY date, id)`) and the `GROUP BY` queries in `server/src/services/summaryService.js`. Monthly category breakdowns exclude transfers (`is_transfer = 0`) since they're internal movement, not real income/spend.
- A second table, `budgets` (one row per month+category, `UNIQUE(month, category)`), holds per-category monthly budget amounts: `month` is a `'YYYY-MM'` string, `category` is one of Spending's outgoing categories (budgeting is **Spending-only** — `budgetService.js` validates via `getBudgetableNames(ACCOUNTS.SPENDING)`, a Savings category of the same name is never a valid budget target), `amount REAL NOT NULL CHECK (amount >= 0)`. Every Spending outgoing category always has a budget value as far as any caller can observe — **the unset-vs-zero distinction was removed from the product/API surface**: a category nobody has ever set behaves identically to one explicitly set to `amount = 0`. Internally the table stays **sparse** (a row exists only once a `PUT` happens — there was no migration and no pre-seeding of zero rows); `getBudgetsForMonth` synthesizes the `0` default on read for any category with no row. So a row vs. no-row still physically differ at the DB level, but that difference is now purely an implementation detail — no caller should query the table to infer "did the user set this?", since that question is no longer product-meaningful. `db.js` applies migrations via four independent existence guards in sequence — a table-existence check for `transactions` (`001_init.sql`), then `budgets` (`002_budgets.sql`), then `categories` (`003_categories.sql`), then a column-existence check (`pragma_table_info('categories')` for `account_id`) for `004_category_accounts.sql` — rather than a single versioned migration runner; there is no migration-version-tracking table yet, so adding a fifth migration means adding a fifth guard in the same pattern.

### Backend structure (`server/src`)
`index.js` wires four route groups under `/api`: `routes/accounts.js`, `routes/transactions.js` (CRUD + the special `/transactions/transfer` endpoint), `routes/summary.js` (`/summary/daily`, `/summary/monthly`), `routes/budgets.js`. Routes delegate to `services/` (`transactionService`, `balanceService`, `summaryService`, `budgetService`), which are the only modules that touch `db.js` directly. `transactionService.js` and `budgetService.js` each export their own `ValidationError` class that routes map to HTTP 400.

- **Budgets API** (`routes/budgets.js` → `services/budgetService.js`): budgeting applies to the **Spending account's categories only** — there is no per-account budgeting. `GET /api/budgets?month=YYYY-MM` returns `{ month, budgets: [{ category, amount }] }`, and the array is now **exhaustive**: every one of `getBudgetableNames(ACCOUNTS.SPENDING)`'s categories is always present, in that function's order (not alphabetical), with `amount` defaulting to `0` for any category that has no row in the `budgets` table — there's no "unset" entry to omit anymore. `PUT /api/budgets` with body `{ month, category, amount }` upserts a single category's budget (`INSERT ... ON CONFLICT(month, category) DO UPDATE`) and returns the saved `{ month, category, amount }`; `amount = 0` is and always was a valid, explicit value. There is no `DELETE /api/budgets` endpoint — it was removed along with the service's `clearBudget` and the client's "Clear" button, since "revert to unset" is no longer a concept consumers can act on; to zero out a category the user sets its budget to `0` instead. Both endpoints validate `month` against `/^\d{4}-(0[1-9]|1[0-2])$/`, `category` against `getBudgetableNames(ACCOUNTS.SPENDING)` (Spending's live outgoing category names), and (for `PUT`) `amount` as a finite number `>= 0`; failures throw `ValidationError` → HTTP 400. There is no dedicated "budget vs. actual" endpoint — the comparison against real spend is done **client-side**, and lives in exactly one place: the Overview dashboard's Budget card (`pages/DashboardPage.jsx`), which derives actuals from the already-fetched, Spending-filtered transactions array rather than calling `/api/summary/monthly` — see the Overview budget card note below. `pages/BudgetPage.jsx` is edit-only now (no comparison chart) but still calls both `GET /api/budgets` and `GET /api/summary/monthly` to show each row's current-month actual spend next to its editable budget input.

- **Import LLM-suggest endpoint** (`routes/imports.js` → `services/importLlmService.js`): `POST /api/imports/suggest` is ASSIST-only — it never replaces or shortcuts the deterministic import pipeline (`parseFile` → Step 2-3 mapping → Step 4 review → `commitImport`'s server-side re-validation), it only produces better defaults for the same `columnMapping`/`categoryMapping`/`accountMapping` prefill seams `guessColumnMapping.js` and the fuzzy matchers already fill. Request body: `{ headers, sampleRows, accountLabels, commentCol, fileHash }` (JSON, not multipart — the file was already parsed by `/parse`); the server builds `knownCategories` itself live from `categoryService.getOutgoingNames/getIncomingNames` across both accounts (not caller-supplied, so a client can't inject a fake vocabulary the LLM could "satisfy"). Response is always HTTP 200 with `{ suggestion: null }` for every expected failure (key unset, network/timeout, non-2xx upstream, malformed/oversized body, schema-invalid) — a real 500 only happens for a genuinely unexpected server error. `importLlmService.js` is the only module that talks to Ollama Cloud; its `buildRequest`/`parseResponse` are deliberately isolated from validation/caching so the wire shape can be corrected in one place. **The wire shape is an ASSUMPTION, not a confirmed contract**: `buildRequest` posts an OpenAI chat-completions-compatible body (`POST {OLLAMA_CLOUD_BASE_URL}/v1/chat/completions`, `Authorization: Bearer {OLLAMA_CLOUD_API_KEY}`, `{ model, messages, response_format: { type: 'json_object' } }`) and `parseResponse` unwraps `choices[0].message.content` as a JSON string — this has not been verified against a real Ollama Cloud call (only mocked transports per the build dispatch); expect to revisit these two functions after the first real-API test. Config: `OLLAMA_CLOUD_API_KEY` (bearer token; feature is hard-off if unset/blank), `OLLAMA_CLOUD_MODEL` (model name string), `OLLAMA_CLOUD_BASE_URL` (must parse as `https:` or the feature is treated as not-configured — no unauthenticated or default-endpoint fallback). `readConfig()` reads `process.env` fresh on every call (not cached at module load). Server enforces, regardless of what the caller sends: a hard 5-row sample cap (`MAX_SAMPLE_ROWS`, not caller-raisable), comment-column redaction keyed off the caller-supplied `commentCol` index (cells replaced with the literal `<redacted>` before the prompt is built), a request timeout (`REQUEST_TIMEOUT_MS`, `AbortController`-based), and a response-size cap (`MAX_RESPONSE_BYTES`) with clean rejection of non-JSON/oversized bodies. `validateSuggestion` is strict allow-list validation (column indices in range, enums checked, category names checked against `knownCategories` unless `isNew`, account ids in `{1,2}`) with **drop-whole-on-any-deviation** semantics — there's no partial-trust/field-repair path. Results are cached in-memory by `fileHash` (caller-supplied, preferably a hash of the full uploaded file's bytes; the route derives a sha256 fallback from `headers`+`sampleRows` if omitted) in a process-lifetime `Map` with no eviction — re-clicking "Suggest" on an unchanged file is free and skips the network entirely. `dotenv/config` is now loaded as the first import in `server/src/index.js` (the server had no `.env` support before this feature); `server/.env` and `server/.env.*` are gitignored (root `.gitignore`) — there is no `.env.example` checked in yet, so a fresh clone needs the three `OLLAMA_CLOUD_*` vars documented/communicated out-of-band until one is added.

### Frontend structure (`client/src`)
Three routed pages (`react-router-dom`) under `App.jsx`: `pages/DashboardPage.jsx`, `pages/TransactionsPage.jsx`, `pages/BudgetPage.jsx` (nav order in `components/layout/Header.jsx`: Overview, Transactions, Budget — there is no standalone Breakdown page/route anymore; its content now lives at the bottom of Overview, see below). All server I/O goes through `api/client.js` (a thin fetch wrapper hitting `/api/...`, proxied by Vite in dev), including `getBudgets`/`setBudget` (there is no `clearBudget` — that call and its "Clear" button were removed along with `DELETE /api/budgets`).

- **Transactions list** (`components/transactions/`): the flat transaction array from the API is grouped client-side into year → month → day via `utils/dateUtils.js` (`groupByYearMonthDay`), rendered by `TransactionList` → `TransactionGroup` → `DayGroup` → `TransactionRow`. Years/months/days display most-recent-first; transactions *within* a day keep the array's original (chronological) order. `DayGroup` computes per-day combined and per-account in/out subtotals plus end-of-day balances (`dayTotals.js`) and only renders the per-account breakdown when a day actually spans more than one account (avoids redundant rows on a single-account day or filtered view).
- **Highlight rules** (`components/transactions/highlight.js`): `highlightClassFor` returns a single flat `'highlight-transfer'` row class for any transfer leg, regardless of direction — there's no direction-specific row coloring. For non-transfer `out` transactions it returns `'highlight-orange'`/`'highlight-red'` (`>$20`/`>$40`), rendered in `TransactionRow.jsx` as a small colored dot next to the amount, not an amount-cell background fill. `transferBadgeFor` provides the direction cue instead: an IN/OUT text badge (green/red) rendered in the comment cell, so the in/out distinction isn't color-only (WCAG 1.4.1).
- **Currency formatting** (`utils/format.js`): accounting-style — `formatCurrency` parenthesizes negative values (no minus sign), `formatInflow`/`formatOutflow` are for non-negative amounts that represent in/out flows — used both for sums (day/account subtotals) and for individual transaction row amounts in `TransactionRow.jsx` (outflows render parenthesized there too, matching the subtotal convention). `components/transactions/AccountingValue.jsx` takes an already-formatted string and splits a trailing `)` into a separate right-edge gutter span (`aria-hidden`) so the last digit lines up across rows regardless of whether a value is parenthesized; `BalanceValue.jsx` is a thin wrapper around it for `formatCurrency` specifically (used for running/end-of-day balances in `TransactionRow.jsx`/`DayGroup.jsx`), while the Transactions OUT column calls `AccountingValue` directly with `formatOutflow`'s output.
- **Daily budget** (`pages/DashboardPage.jsx`, Overview only): the "Budget $50.00/day" pill in the "Daily insights" card header is click-to-edit — clicking swaps it for a number input plus a "No budget" clear button (shown only when a budget is currently set). Backed by `hooks/useDailyBudget.js`, a `localStorage`-only hook (key `ledger.dailyBudget`, no backend model) returning `[dailyBudget, setDailyBudget]` where the value is `number | null`; `null` is persisted as the literal string `'null'` (distinct from "key absent," which defaults to `constants/budget.js`'s `DEFAULT_DAILY_BUDGET = 50` for first-time users) so an explicit clear survives a reload instead of reverting to the default. Invalid input (negative, non-numeric, empty) on commit reverts to the last valid value rather than crashing or saving garbage. When unset, `utils/insights.js`'s budget-relative figures (spent-today over/under, daily average target, days-on-budget) all degrade to neutral text ("—" / "No daily budget set") instead of NaN/undefined, and the Monthly insights daily bar chart drops its `.budget-line` reference marker entirely and stops red-coloring "over" bars.
- **Forms**: transaction/transfer entry is a single `components/transactions/TransactionModal.jsx` (normal + transfer modes). Editing an existing transaction is inline, in the row itself, not a separate panel: `TransactionRow.jsx` renders either a `ReadRow` or an `EditRow` depending on whether its id matches `editingId`, switched by clicking the pencil icon. `pages/TransactionsPage.jsx` holds `editingId` (number|null, not a copy of the transaction object) and threads `editingId, onEdit, onSaveEdit, onCancelEdit, onDelete` down `TransactionList` → `TransactionGroup` → `DayGroup` → `TransactionRow` (same threading shape as the rest of the list); `onEdit` is called with the row's id. `EditRow` turns Date (inline in the Comment cell, alongside the comment input — there's no dedicated Date column), Amount (whichever of the in/out cells matches the row's `direction`; the other stays `—`), Category (a `<select>` populated from `outgoingFor`/`incomingFor` for the row's account+direction via `CategoriesContext`), and Comment into inputs; Account, the computed running balance, and the inactive in/out cell are never editable. For a transfer row, Category renders as plain text (locked, no select) and the save payload omits `category` entirely, since the backend rejects category changes on transfers — date/amount/comment still edit both linked legs atomically. Save/Cancel (lucide `Check`/`X`) replace the pencil/trash icons in the Actions cell while a row is editing; `Enter` saves, `Escape` cancels; only one row can be in edit mode at a time. Validation/API errors render inline as a `role="alert"` row directly beneath the editing row, not a page banner. On the backend, `updateTransaction` in `server/src/services/transactionService.js` accepts `{ date, amount, comment, category }`: for a normal transaction a provided `category` is revalidated via `isValidNormalCategory` (and rejected if it's a system transfer category); for a transfer row, *any* provided `category` throws `ValidationError` ('category cannot be changed on a transfer'); the linked leg's `UPDATE` only ever touches `date`/`amount`/`comment`, never `category`, so each leg keeps its own `transfer-in`/`transfer-out`. Omitting `category` preserves the existing value. `PUT /transactions/:id` (`routes/transactions.js`) is a thin pass-through of `req.body` and required no route changes.
- **Modal portal target**: `TransactionModal.jsx` (and its nested `CategoryManagerModal`, rendered as its JSX child) render via `createPortal(..., document.getElementById('modal-root') || document.body)` into a dedicated `<div id="modal-root" />` that `App.jsx` renders as a sibling of `<main>` — inside the themed `.app-root` tree (so theme CSS custom properties still cascade in) but *not* a descendant of any page's `.page-animate` wrapper. This matters because `.page-animate` has a CSS `animation` whose keyframes include a `transform`, and browsers treat that as establishing a containing block for `position: fixed` descendants — without the portal, the modal would be positioned relative to that tall, scrolled page wrapper instead of the viewport (visible as the modal opening far below the fold on a scrolled-down page). Any new full-page modal should portal to `#modal-root` the same way, not render inline inside a page's component tree.
- **Breakdown section** (folded into `pages/DashboardPage.jsx`, below the Top spending/Recent activity row on the Overview page — there is no standalone Breakdown page/route): a "Breakdown" heading row, then Monthly insights (money in/out/net + the daily spending bar chart), then two stacked, separately-labeled sections — "Spending" and "Savings" — each rendered via a module-level `DonutCard` component as a Money out/Money in pair of `components/breakdown/DonutChart.jsx` donuts (hand-rolled SVG, no charting library) plus `CategoryBarList.jsx`, followed by a module-level `InOutCompareCard` showing a two-bar (Money in `var(--green)` / Money out `var(--red)`) comparison of the account's monthly totals with `formatCurrency` value labels under each bar (not color-only). The "Breakdown" heading row holds two controls side by side: a `MonthSwitcher` (the same component `BudgetPage.jsx` uses) and `BreakdownControls.jsx`'s cash value/percentage/both toggle (at least one of the two must stay checked); the toggle applies to all four donuts at once (it does not affect the in/out bars, which have no percentage mode). The `MonthSwitcher` is backed by its own decoupled `selectedMonth` state (initialized to `currentMonthStr()`) and its own `breakdownTransactions` fetch (`monthRangeFor(selectedMonth)` via `api.getTransactions`), entirely independent of the page's current-month `monthTransactions`/`loadAll` fetch — so it scopes Monthly insights *and* both breakdown halves (donuts, category bar lists, `InOutCompareCard`) to whichever month is selected, while everything else on Overview stays anchored to the real current month. Unlike the rest of Overview (which is Spending-only — see below), this section **always shows both accounts**, same treatment as Net worth. A module-level `breakdownFor(transactions, accountId, direction, colorFor)` helper derives each of the four category breakdowns **client-side** from `breakdownTransactions` — filtering by `account_id`/`direction`, excluding transfers (`is_transfer`), grouping by category, and sorting descending; `InOutCompareCard`'s two totals are simply the sum of each account's `breakdownFor('in'/'out')` array, so they inherit the same transfer-exclusion as the donuts above them rather than recomputing independently. Bars scale to whichever of the two totals is larger, with a zero-safe floor (4% minimum height) so an account with no activity in one direction still renders a visible flat bar instead of a zero-height/NaN one.
- **LLM-assisted import suggestion** (`components/imports/ImportSuggestAI.jsx`, rendered at the top of `ImportModal.jsx`'s Step 2/Columns): an opt-in "Suggest with AI" button — ASSIST only, same gate as the deterministic path. Clicking it does not send anything; it opens a consent panel naming Ollama Cloud explicitly and rendering the exact payload (headers, up to 5 sample rows with the comment column shown as `<redacted>`, distinct account-column labels) via the same `buildPayload()` that builds the real request, so the preview is byte-for-byte what would be sent. Only "Confirm and send" calls `api.suggestImportMapping`. A successful, shape-checked response (`suggestion.columnMapping` truthy, `categoryMapping`/`accountMapping` both arrays) is merged into state via `ImportModal.applyAiSuggestion`: it overlays `suggestion.columnMapping` onto the existing Step 2 `columnMapping` (explicitly re-pinning `fixedAccountId`, which the LLM contract never proposes) and stashes the raw `categoryMapping`/`accountMapping` arrays in `state.aiCategorySuggestion`/`aiAccountSuggestion`; `proceedToValues()` only converts those into the `Map`s Step 3 uses if the user hasn't already started manually editing (`categoryMapping.size > 0`/`accountMapping.size > 0` takes precedence, same rule the deterministic prefill already followed). Any null/failed/malformed response is a silent no-op — the deterministic `guessColumnMapping` + fuzzy category/account prefill already populated underneath is left untouched, and a small `role="status"` note ("AI suggestion unavailable — using local mapping.") appears instead of an error. The wizard is never blocked on this call; Next on Step 2 works with or without it.

- **Budget page** (`pages/BudgetPage.jsx`, route `/budget`): edit-only — the old read-only "Budget vs. actual" comparison chart has been removed (that comparison now lives solely in the Overview Budget card, see below). A `MonthSwitcher` (`components/layout/MonthSwitcher.jsx` — the shared month-navigation control used identically on all three tabs: prev/next chevron buttons stepping over `dateUtils.js`'s `prevMonthStr`/`nextMonthStr`, plus a native `<input type="month">` in the middle for jumping directly to any month; same `{ month, onChange }` props at every call site, where `onChange` receives the `'YYYY-MM'` string. Every month is navigable since budgets aren't constrained to months with transaction history) drives an editable "Category budgets" list, one row per Spending's outgoing categories (`outgoingFor(ACCOUNTS.SPENDING)` from `CategoriesContext`, in that fixed order — not sorted by spend; colors via `colorFor(ACCOUNTS.SPENDING, category)`). Each row still shows actual spend (fetched via `GET /api/summary/monthly`) next to the editable budget input, and commits on blur/Enter via `PUT /api/budgets` (no Save button); there is no "Clear" button anymore — every category always has a numeric budget (defaulting to 0), so an emptied input commits `0` rather than reverting to an unset state; per-row errors render inline, not as a page banner. Health classification (`healthFor` in `BudgetPage.jsx`) is `under` (<90%) / `near` (90–100%) / `over` (>100%) only — there's no `unset` branch anymore, so a `$0` budget with any spend reads `over` by design — with `near`/`over` always paired with a non-color text suffix ("— near limit" / "— over"), matching the `highlight.js` convention of not relying on color alone.
- **Overview is Spending-only** (`pages/DashboardPage.jsx`): there is no account selector on this page — `scopeAccountId` is a fixed constant (`ACCOUNTS.SPENDING`), not state. Daily insights, the budget card's actuals, Top spending, and Recent activity are all derived from `monthTransactions` filtered to Spending only; the page no longer calls `api.getMonthlySummary` at all (removed from `loadAll`). Monthly insights (money in/out/net + the daily bar chart) is the exception — it now lives inside the Breakdown section and is driven by that section's `selectedMonth`/`breakdownTransactions`, not the current-month `monthTransactions` (see the Breakdown section note below). The balance-cards row shows the Spending balance card plus the combined Net worth card. Net worth, the budget card (entirely — it's Spending-only by definition, not just by happening to match the page default), and the bottom Breakdown section (which always shows both accounts) are the only places Savings data appears on this page at all — Savings has no other representation on Overview; its own insights live on the Transactions/Budget pages and the Breakdown section's "Savings" half. Each Recent activity row shows only the category's color dot (no category-name text beside it) plus the comment and amount — a transfer row's dot is neutral rather than category-colored.
- **Overview budget card** (`pages/DashboardPage.jsx`): a summary card between "Daily insights" and the account balance row, scoped to **Spending** (budgeting doesn't apply to Savings) — total budgeted vs. total spent across **every** Spending outgoing category, since all of them now always have a budget value; categories nobody has explicitly set contribute `0` to "Total budgeted" but their actual spend still counts toward the under/near/over classification (so a never-budgeted category with any spend reads `over`), plus a **full** worst-first comparison list (`over` before `near`) with a color legend (`--green`/`--warning`/`--red` swatches for under/near/over) — this absorbed the Budget page's old comparison chart, so there is now exactly one place to see budget vs. actual, and the list is no longer capped at 3 rows (no "+N more" line). There is no longer a distinct "no budgets set" empty state — since every category always carries a budget value, the card always has rows to show.
- **Category source of truth**: categories are no longer a static shared constant — they're **per-account** rows in the `categories` table (see the data-model note above), fetched live via `CategoriesContext` (`contexts/CategoriesContext.jsx`), which exposes `outgoingFor(accountId)`, `incomingFor(accountId)`, and `colorFor(accountId, name)`. Components call these with `ACCOUNTS.SPENDING`/`ACCOUNTS.SAVINGS` rather than importing a fixed category list. `ACCOUNTS`/`ACCOUNT_NAMES` (the fixed account ids/names, not categories) still live in `constants/categories.js` on both client and server and must stay in sync the same way as before.
- **Category color spacing**: the 10 non-system, non-travel seed categories' colors (`server/src/migrations/003_categories.sql`, re-applied to existing dev DBs by `005_recolor_categories.sql`) sit at hues spaced 36° apart (fixed sat 48%/light 58%) so no two read as the same color, including across outgoing/incoming (e.g. `income` no longer duplicates `food`'s hex). `travel` keeps its neutral gray `#8A8F98`, intentionally shared with the system `transfer-in`/`transfer-out` colors since travel and transfer categories are never shown in the same picker. `services/categoryService.js`'s `assignColor()` extends this to future user-added categories once the 8-slot `PALETTE` is exhausted: it searches hue space for a hue at least 30° from every color already in use on that account (not just an exact-hex check), falling back to the best available hue if an account gets crowded enough that no hue clears the threshold.

### Styling
Single global stylesheet `client/src/index.css` using CSS custom properties defined in `:root` — a warm dark theme modeled on the Claude app (charcoal background, terracotta accent `--accent`) with a small type scale (`--font-xs` … `--font-2xl`) and semantic color tokens (`--money-in`, `--money-out`, `--topup`, `--transfer-out`, etc.). Icons are from `lucide-react`, sized 14–16px, colored via `currentColor` so they inherit theme colors. Buttons/inputs/selects need an explicit `font-family: inherit` rule (added in `index.css`) since browsers don't inherit page font-family into form controls by default — if fonts look inconsistent after future styling changes, check that rule first before assuming a font-loading issue. Theme tokens live in `client/src/contexts/theme.js` (three sets: refined/dark, editorial/light, terminal/mono); `--warning` is for decorative fills (bar segments, dots) while `--warning-text` is a separate token specifically for near-budget TEXT — on the editorial (light) theme `--warning`'s amber fails WCAG AA as body text on light surfaces, so `--warning-text` is remapped to a darker amber there (the other two themes keep `--warning-text` identical to `--warning`). Any new near-budget-style text should use `--warning-text`, not `--warning`.

When changing colors or highlight rules, keep `server/src/constants/categories.js` and `client/src/constants/categories.js` in sync (account ids/names only now — category lists themselves live in the DB, per account, not in these constants files) — there is no shared package between client and server.

## Known security risks

- **`xlsx` (SheetJS) `@0.18.5`, npm registry — accepted, unpatched advisories.** Used by `server/src/services/importService.js`'s `parseFile` to parse uploaded CSV/.xlsx bytes for the transaction import feature. `npm audit` reports two unpatched high-severity advisories with "no fix available" on npm: Prototype Pollution (GHSA-4r6h-8v6p-xvw6) and ReDoS (GHSA-5pgg-2g8v-p4x9) — SheetJS only publishes patched builds via their own CDN (cdn.sheetjs.com), not the npm registry. **Accepted as-is** because this is a single-user local app: the only untrusted input reaching `xlsx` is the user's own uploaded file, processed against their own local DB — there is no multi-tenant or network-exposed attack surface, and the advisories grant no privilege beyond what the local user already has. **Reopens as a blocker** if any of: (1) the app becomes multi-user or network-exposed, (2) the import endpoint ever ingests files from anyone but the local user, or (3) a registry fix becomes available (re-run `npm audit` periodically to check). Exit path at that point: pin SheetJS's CDN-hosted patched build, or move to a CSV-only import path via a maintained CSV parser.

- **LLM-assisted import ("Suggest with AI") — first and only path by which app data leaves the user's machine to an external third party (Ollama Cloud).** Every other feature in this app (parse/map/review/commit, the rest of the import wizard included) runs 100% locally against the user's own SQLite file; this is the one deliberate exception, and it changes the threat model for the data it touches, so the conditions below are what keep it acceptable:
  - **Opt-in, per-file, with a mandatory consent/preview step — not on by default, not a one-time global toggle.** Nothing is sent until the user clicks "Suggest with AI" (`client/src/components/imports/ImportSuggestAI.jsx`, Step 2 of the import wizard) *and* then explicitly clicks "Confirm and send" on a panel that names Ollama Cloud as an external third party and renders the literal payload about to be sent (not an approximation — the preview is built by the same `buildPayload()` function that constructs the real request body).
  - **Minimal payload, server-enforced regardless of what the client sends:** headers, up to 5 sample rows (`MAX_SAMPLE_ROWS` in `server/src/services/importLlmService.js`, hard-capped server-side and not caller-raisable even if the client sent more), the comment/memo column redacted to the literal string `<redacted>` before the prompt is built (keyed off the client-supplied `commentCol` index — both client preview and server payload redact the same way, from the same source value, so they can't drift), and the app's own category vocabulary (`knownCategories`, built server-side from `categoryService`, not client-supplied). No full file, no full transaction history, no API key, ever leaves in the request.
  - **The API key (`OLLAMA_CLOUD_API_KEY`) is strictly server-side.** Read only in `server/src/services/importLlmService.js` via `process.env`, never returned in any response, never logged (`console.error` paths in `routes/imports.js`'s `handleError` stay generic), and never shipped to the browser (not a `VITE_`-prefixed var, not in any client bundle). The feature is hard-disabled — `isConfigured()`/`readConfig()` return falsy and `POST /api/imports/suggest` resolves `{ suggestion: null }` — whenever `OLLAMA_CLOUD_API_KEY`, `OLLAMA_CLOUD_MODEL`, or `OLLAMA_CLOUD_BASE_URL` is unset/blank, or the base URL isn't `https:`; there is no unauthenticated or default-endpoint fallback.
  - **The LLM's response is strictly schema-validated, drop-whole on any deviation.** `validateSuggestion` in `importLlmService.js` allow-lists every field (column indices in range, fixed enums, category names checked against the server-built vocabulary unless explicitly flagged `isNew`, account ids in `{1,2}`); any single invalid field discards the entire suggestion rather than partially trusting it. The response only ever **prefills** the same `columnMapping`/`categoryMapping`/`accountMapping` state the deterministic guessers (`guessColumnMapping.js`, fuzzy matchers) already populate — it never reaches the commit path directly. The existing Step 4 review gate (every flagged row surfaced, `commitImport`'s server-side re-validation of every row inside one atomic transaction) is unchanged and remains the actual integrity boundary; the LLM suggestion is exactly as overridable, and exactly as easy for the user to ignore or correct, as a wrong fuzzy-match guess is today.
  - **Any failure mode degrades silently to the pre-existing deterministic prefill.** Timeout (server-enforced via `AbortController`, `REQUEST_TIMEOUT_MS`), network error, non-2xx upstream response, oversized/non-JSON body, or schema-invalid content all collapse to the same `{ suggestion: null }` signal from the server; the client (`ImportSuggestAI.jsx`) treats that identically to a thrown fetch error — `onResult(null)` is a no-op, the wizard's existing `guessColumnMapping` + fuzzy category/account prefill is left completely untouched, and a non-alarming `role="status"` note is shown instead of an error. The wizard is never blocked on this call.
  - **Accepted as-is** under the current threat model: single-user, local app, opt-in per file, minimal/redacted payload, no auto-commit. **Reopens for re-review** if any of: (1) the app ever becomes multi-user or network-exposed (egressing one user's data on another's action is a materially different risk), (2) auto-commit or any "auto-apply AI suggestion without review" path is ever added (this would remove the only backstop that currently contains a maximally-wrong LLM response to "a bad prefill the user must still catch," not a bad write), (3) the per-file opt-in/preview gate is ever weakened to an always-on default or a buried settings toggle, or (4) the sample-row cap, redaction, or schema-validation drop-whole behavior is loosened.
